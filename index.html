1. Introduction & Project Goal

This document outlines the architecture and functionality of the Student Management System, a console-based application written in C++. The primary goal of this project is to provide a simple yet functional system that allows for two distinct user roles:

  • An Admin who can view all student records and manage their own password.
  • A Student who can register, log in, view, and edit their own personal information.

2. Project Structure & File Overview

The project is organized into multiple files, each with a specific responsibility. This separation of concerns makes the code easier to read, manage, and debug.

admin.cpp (Main Entry)

Purpose: To be the single starting point of the application. Its main() function simply calls selectUser() to kick off the user interaction.

selectUser.cpp

Purpose: To direct the user down the correct path. It displays the first menu, asking if the user is an "Admin" or a "Student", and calls the corresponding login function (`adminlogin()` or `studentlogin()`).

adminlogin.cpp

Purpose: To guard the admin-only sections. It prompts for an admin username and password, reads credentials from admin_cred.txt, and grants access only if they match. On success, it calls studentDetails().

studentlogin.cpp

Purpose: To act as the main hub for students. It provides two options:

  • Log in as an existing user, which involves reading students.txt to find the user and verify a password.
  • Register as a new student, which calls the newStudent() function.

newStudent.cpp

Purpose: To handle the creation of new student accounts. It collects the student's details and appends them as a new line to students.txt using ofstream with the ios::app flag to avoid overwriting data.

studentDetails.cpp

Purpose: To serve as the admin's main control panel. After a successful admin login, this function displays a menu with options to view student details, change the admin password, or log out.

student.cpp

Purpose: To provide a complete view of all student data for the admin. It reads and displays every field for every student from students.txt.
Feature Highlight: This file uses a try-catch block to handle potential errors, such as the students.txt file not being found. This prevents the application from crashing and instead shows a user-friendly error message.

viewstudent.cpp

Purpose: To provide a quick, summarized view of student data. It reads from students.txt but only displays the Name, Roll No, and Email for a more concise list.

studentInfo.cpp

Purpose: To give students control over their own data. Once a student logs in, this function displays their current information and a menu to edit each field.
Feature Highlight: It uses a helper function, updateFile, which safely updates the student's record by writing all data to a temporary file (temp.txt), deleting the original students.txt, and then renaming the temporary file.

changeAdminPassword.cpp

Purpose: To allow the administrator to securely change their password. It first verifies the current password, then prompts for a new one and checks its strength (requiring an uppercase letter, a number, a special character, and minimum length). It then updates admin_cred.txt.

3. Core C++ Features & Libraries

The application is built entirely with standard C++ libraries, ensuring it is portable and does not require external dependencies. The following features are key to its operation:

File Handling with <fstream>

This is the most critical library for the project, as it handles data persistence. It allows the program to read from and write to files, which act as a simple database.

  • ifstream: Used for input (reading from files). We use it in adminlogin.cpp and studentlogin.cpp to verify credentials and in student.cpp to display data.
  • ofstream: Used for output (writing to files). We use it in newStudent.cpp to register new users and in studentInfo.cpp to save updated details. The ios::app mode is used to ensure new students are added without deleting existing ones.

Input/Output with <iostream>

The fundamental library for console interaction. cout is used to display all menus, prompts, and information to the user. cin is used to capture all user input, such as menu choices and login credentials.

Error Handling with try-catch

This is a C++ mechanism for building robust and crash-proof code. It works like a safety net: we try a piece of code that might fail (like opening a file). If an error occurs, the program doesn't crash. Instead, the error is caught, and we can execute code to handle it gracefully, like showing a message "File not found." This is implemented in student.cpp to ensure the program remains stable even if students.txt is missing.

File System Management with <cstdio>

While most file I/O is done with <fstream>, the C Standard I/O library provides essential file management tools. Specifically, the remove() and rename() functions are used in studentInfo.cpp to implement a safe update procedure. The old file is only deleted after the new one with the updated information has been successfully created.

String and Character Handling (<string>, <cctype>)

The std::string class is used everywhere to manage text data like names and emails. The <cctype> library is used in changeAdminPassword.cpp to check if characters in the new password are uppercase, digits, or punctuation, enabling the password strength checker.

4. Data Storage

The application uses a simple flat-file database system. All data is stored in plain text files, making it easy to view and manually edit if needed.

students.txt

  • Purpose: Stores all student records.
  • Structure: Each line represents a single student record.
  • Format: Data fields are separated by a single space.
  • Order: Name Username Roll_No Email Age

Example:

admin_cred.txt

  • Purpose: Stores the administrator's credentials.
  • Format: username password

5. User Flow Diagrams

Admin Path

Student Login & Edit Path

main() → selectUser() → studentlogin() → studentInfo() → (Edit Info → updateFile()) → Logout

Student Registration Path

main() → selectUser() → studentlogin() → newStudent()

6. How to Compile and Run

The project is split into multiple .cpp files, which need to be compiled and linked together to create a single executable program. This can be done using a C++ compiler like g++. Open your terminal or command prompt, navigate to the project directory, and run the following command:

g++ admin.cpp studentlogin.cpp studentDetails.cpp adminlogin.cpp student.cpp selectUser.cpp viewstudent.cpp studentInfo.cpp newStudent.cpp changeAdminPassword.cpp -o app && ./app

Command Breakdown:

  • g++: This invokes the GNU C++ compiler.
  • admin.cpp studentlogin.cpp ...: This lists all the source code files that make up the project. The compiler needs all of them to build the final program.
  • -o app: The -o flag tells the compiler to name the output executable file app. Without this, the default name is usually a.out.
  • &&: This is a shell operator that chains commands. It means: "If the first command (compilation) succeeds, then execute the second command."
  • ./app: This command runs the executable file app that was just created. The ./ is necessary to tell the shell to look for the program in the current directory.

7. Challenges & Solutions

During the development of this project, several challenges were encountered. This section documents the key problems and the strategies used to solve them.

Challenge: Compiling and Linking Multiple Files

Problem: A significant initial hurdle was figuring out how to compile all the separate .cpp files into a single, working application. It was not immediately clear how to make functions from one file available to another.

Solution: The solution was to use the g++ compiler to link all source files in a single command. By providing all .cpp files as input to the compiler, it can resolve the function calls between them and build one cohesive executable. The final command structure (g++ [all .cpp files] -o [output_name]) was key to overcoming this.

Challenge: Implementing the "Edit User" Feature

Problem: Safely updating a specific student's information in the students.txt file was complex. Simply overwriting the file would erase all other student records.

Solution: This was solved by implementing a temporary file strategy, found through research on Google and with assistance from AI tools. The process involves reading the original file line-by-line, writing to a new temp.txt file, substituting the updated information when the correct user is found, and then replacing the original file with the temporary one. The <cstdio> library, with its remove() and rename() functions, was essential for this.

Challenge: Securely Changing the Admin Password

Problem: The feature to change the admin password needed to be secure. It required not only verifying the current password but also encouraging the use of a strong new password.

Solution: Code from a previous academic assignment was adapted for this purpose. The solution involves checking the new password against a set of criteria (presence of uppercase letters, numbers, and special characters) using functions from the <cctype> library. This adds a layer of security by guiding the admin to create a more robust password.

General Approach to Problem-Solving

A recurring solution for many issues was the use of online research (Google) and modern AI coding assistants. For file-related operations that could fail (like a missing file), the try-catch error handling block was identified as a best practice, as seen in student.cpp.

8. Conclusion & Future Improvements

The Student Management System successfully implements core CRUD (Create, Read, Update) functionality in a simple console environment. It features a clear separation of roles between an administrator and students, and includes key features like secure admin password management with strength checking, and robust file handling.

Potential Future Improvements

  • Individual Student Passwords: Modify the students.txt file to include a unique, hashed password field for each student instead of a global password.
  • Input Validation: Add checks to ensure usernames are unique upon registration and that data like email is in a valid format.
  • Delete Functionality: Allow an admin or a student to delete a student record.
  • Robust Input Error Handling: Improve robustness by handling incorrect input types (e.g., entering a letter when a number is expected), which currently can crash the application.
